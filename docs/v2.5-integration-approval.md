# Auto-G16 v2.5 Integration Approval Contract

## Purpose

`gaussian-v25-integration-review/1` is the non-authorizing owner overlay for
the v2.5 decision, method-evidence, TS-seed, closure-priority, and execution-
batch slices. It exists because each component intentionally validates only
its own artifact. The overlay replays every owner validator and closes the
cross-Skill references without changing any owner schema or implementation.

The overlay always fixes `calculation_ready: false`, `executable: false`,
`no_submission_authorization: true`, `no_automatic_promotion: true`, and
`live_actions: false`. It requires a separate exact input review and a fresh
live approval for every physical attempt. Neither the overlay nor any one of
its bound artifacts can render an input, reserve an attempt, submit, retry,
cancel, clean up, change chemistry, or expand a search.

## Exact owner chain

1. The exact `auto-g16-method-evidence-brief/1` must be present in the
   `gaussian-mechanism-discussion/1` evidence set. The discussion must contain
   the declared `method` claim as explicit user input; an AI recommendation or
   a reviewable evidence brief is never a method decision.
2. Every selected closure route must cite the exact mechanism discussion in
   both the mechanism/active-state and user-confirmation hard gates. Its method
   hard gate must cite both the evidence brief and the explicit discussion.
   Every hard gate must pass before the route can enter the overlay.
3. Each selected route binds one exact `gaussian-ts-seed-portfolio/1`. Its
   target must equal the closure route ID, and the closure
   `initial_guess_quality` provenance must be exactly that portfolio token.
   A seed remains an initial-guess hypothesis, not a TS or execution approval.
4. The overlay maps every calculation-bearing node of each selected closure
   bundle (`ts_freq`, both IRC directions, and conditional endpoint Opt/Freq)
   one-to-one to an immutable execution-batch review task. Review and ledger
   task sets must match exactly and remain within the ten-task cap.
5. Each execution identity replays the discussion confirmation scope, the
   explicit method-protocol hash, and a deterministic closure-node objective.
   The TS/Freq task additionally binds the primary seed geometry. The
   initialized ledger must bind the exact immutable batch review.

This mapping records the reviewed plan inventory. Dependency outcomes still
apply: an IRC attempt cannot be approved before accepted TS/Freq and intended-
mode evidence, and endpoint work cannot be approved before both IRC directions
and structural endpoint identification. The integration artifact does not
turn later conditional nodes into immediately runnable work.

## Offline use

All paths are package-root-relative and all bound files are regular,
non-symlink files below one package root:

```bash
./scripts/python core \
  skills/auto-g16-reaction-workflow/scripts/v25_integration.py \
  finalize --root review-package integration.draft.json \
  --output integration-review.json

./scripts/python core \
  skills/auto-g16-reaction-workflow/scripts/v25_integration.py \
  validate --root review-package integration-review.json
```

Finalization refuses overwrite. Validation replays file hashes, owner payload
hashes, the mutable ledger hash, all six bound artifacts through their five
owner modules, selected routes, node/task identity, and the closed authority
constants. It performs no network or live action.
